🧭 dev-harness-plus — 진행 흐름 & 디렉토리 산출/참조 맵

기획 → 설계 → 구현 파이프라인의 각 단계가 어떤 디렉토리를 참조(read)하고 어떤 산출물을 생성(write)하는지 시각화. 좌측은 파트별 입력 기준 폴더(없으면 스스로 진행).

참조(read) · 입력 기준 산출(write) · 생성물 진행 흐름 기획 설계 구현 마무리

🔎 흐름 단계별 세부 설명

P0Discovery + 컨텍스트 · 리스크 게이팅

담당: 메인 대화(PM)

  • 실행 모드 판별 — docs/** 와 소스 트리를 훑어 초기 실행 / 부분 재실행 / 새 실행을 구분한다.
  • greenfield vs brownfield — 기존 코드가 있으면 구조·스택·제약을 감사해 docs/pm/codebase-audit.html로 남기고, 이후 모든 위임에 "기존 코드 정합" 요구를 포함한다.
  • 리스크 기반 트랙 선택 — 변경의 리스크/규모로 경량 / 표준 / 엄격 트랙을 고른다. 고위험(인증·결제·개인정보·외부연동·마이그레이션)은 엄격 트랙.
  • 변경 이력 기록(기존 산출물이 있을 때) — 이번 작업이 추가/수정/삭제이면, 변경이 확정될 때마다 docs/pm/change-log.html 에 날짜·유형·대상·사유·영향을 누적 기록한다(삭제는 사유 필수). 최초 생성에는 생략.
참조docs/** · 소스 트리산출docs/pm/codebase-audit.html (brownfield) · docs/pm/change-log.html (변경 시)

P1기획

담당: product-planner (생성) → reviewer (검증)

  • docs/Planning/ 의 룰·템플릿·정책을 기준으로 PRD·기능명세·사용자 스토리·수용 기준을 작성한다. (폴더가 없으면 모범 사례로 자율 진행)
  • 추적성 매트릭스 docs/traceability.html 에 요구 ID 행을 생성한다.
  • 검증·반영 — reviewer가 추적성·수용기준·범위·리스크 체크리스트로 검증 → FAIL이면 반영 항목과 함께 재호출(최대 2회). 엄격 트랙은 적대적 검증(독립 2회·둘 다 PASS).
  • 통과 조건 — 개발 범위 확정 → 설계로.
참조docs/Planning/(기준) · 기존 docs/산출docs/ · docs/traceability.html · docs/reviews/

P2아키텍처 설계

담당: chief-architect (생성) → reviewer + security-reviewer (검증)

  • docs/Design/ 의 설계 원칙·디자인 시스템·컨벤션을 기준으로 시스템+소프트웨어(추상화)+디렉토리 구조와 ADR을 설계한다. 신뢰 경계·인증/인가를 설계서에 내장한다.
  • 보안 검증(엄격 필수) — security-reviewer가 위협 모델링(신뢰 경계·시크릿·민감데이터)을 수행, 치명/높음 항목을 반영시킨다.
  • 검증·반영 — reviewer가 추적성·근거(ADR)·디렉토리 일관성을 검증 → 반영. 추적성 매트릭스에 요구→설계 결정 매핑을 갱신한다.
  • 통과 조건 — 설계 확정 → 개발 인계로.
참조docs/Design/(기준) · docs/(기획)산출docs/architecture/ · docs/reviews/ · docs/traceability.html(갱신)

P3개발 인계 (실행 모드 전환)

담당: 메인 대화(PM)

  • 사용자에게 "설계 완료, 개발 시작" 의사를 확인한다.
  • 실행 모드 전환 — 서브에이전트 파이프라인 → 에이전트 팀(team-dev). 세션당 팀은 1개만 활성화하므로 설계가 끝난 뒤 소집한다.
산출없음 — 모드 전환 단계

P4개발 로드맵

담당: tech-lead (생성) → reviewer (검증)

  • docs/Development/ 룰을 기준으로, 설계의 모듈·의존성을 순서 있는 구현 단계로 분해해 docs/roadmap.html 를 작성한다.
  • 검증·반영 — reviewer가 설계 정합성·단계 순서(의존성)·완료 기준을 검증 → 반영.
참조docs/architecture/ · docs/Development/(기준)산출docs/roadmap.html · docs/reviews/

P5단계별 구현 (다면 검증 루프)

담당: tech-lead 주관 — devops · backend · frontend + qa · code-reviewer · security-reviewer · debugger

  • 착수(DoR)→계획→구현 — DoR 확인 후 세부 계획을 reviewer가 검증, 통과하면 devops 골격 → backend·frontend가 API 계약 합의 후 병렬 구현한다.
  • 다면 검증(병렬) — qa(동작·경계면) · code-reviewer(코드 품질) · security-reviewer(보안 게이트)를 동시에 돌려 사각을 줄이고, 각 지적을 담당 개발자에 회부해 반영한다.
  • 난항 버그 — 원인 불명·간헐적이면 debugger가 가설→검증으로 근본 원인을 규명한다.
  • 통과·차단 — DoD 충족 시 tech-lead가 통합하고 추적성을 갱신해 다음 단계로. 보안 게이트 치명/높음은 미해결 시 진행 차단.
참조docs/Development/ · docs/Design/ · docs/architecture/산출소스 코드 · .env.example · docs/reviews/ · docs/traceability.html(갱신)

P6전체 검증 → 최종 마무리

담당: qa · reviewer · security-reviewer · (code-reviewer) · tech-writer

  • 전체 통합 검증(병렬) — qa(전체 빌드·핵심 시나리오) · reviewer(수용 기준 + docs/traceability.html 빈 칸 점검) · security-reviewer(최종 게이트).
  • 문서화 — 필요 시 tech-writer가 README·사용/개발 가이드를 코드와 동기화해 작성한다.
  • 마무리 — 통과 시 산출물 경로·구현 요약·추적성 상태·(기존 산출물 변경 시) docs/pm/change-log.html 이력 요약·남은 리스크를 사용자에게 보고하고, 필요 시 팀을 정리한다.
참조docs/traceability.html · 전체 산출물산출README · 사용/개발 가이드 · docs/traceability.html(완성)

📂 디렉토리 지도 (전체)

디렉토리 / 파일구분역할생성·갱신 단계
docs/Planning/입력기획 파트 기준(룰·템플릿·정책). 없으면 planner가 스스로 진행사용자가 사전 배치
docs/Design/입력설계·UI 파트 기준(설계 원칙·디자인 시스템). 없으면 스스로 진행사용자가 사전 배치
docs/Development/입력개발 파트 기준(코딩 표준·테스트·보안 룰). 없으면 스스로 진행사용자가 사전 배치
docs/Templates/입력산출물 HTML 템플릿(*.html). 있으면 그 스타일로, 없으면 표준 다크 스타일로 생성사용자가 사전 배치(선택)
docs/pm/산출PM 산출물 — 코드베이스 감사·회의록·진행보고Phase 0 / 4
docs/pm/change-log.html산출변경 이력 — 기능 추가/수정/삭제의 날짜·대상·사유·영향 (기존 산출물이 있을 때)재실행·변경 확정 시마다
docs/산출기획 산출물 — PRD·기능명세·사용자 스토리Phase 1
docs/architecture/산출설계 산출물 — 설계서·ADR·API 계약Phase 2 (구현 중 계약 갱신)
docs/roadmap.html산출개발 로드맵 — 순서 있는 구현 단계Phase 4
docs/reviews/산출검증 보고 — reviewer·qa·code-reviewer·security-reviewer 결과모든 검증 단계 (1·2·4·5·6)
docs/traceability.html산출추적성 매트릭스 — 요구→설계→구현→테스트Phase 1 생성 · 2·5 갱신 · 6 완성
소스 코드 / .env.example산출실제 구현물 — 모듈·API·UI·골격Phase 5
README / 가이드산출사용자·개발자 문서 (tech-writer)Phase 6
※ 좌측 파트 기준 폴더(docs/Planning · docs/Design · docs/Development)는 입력 기준 문서이며, 존재하지 않으면 해당 파트가 외부 기준 없이 스스로 진행합니다. 산출물 경로와는 별개입니다.
※ 각 화살표(생성/검증/반영) 단위마다 reviewer·qa 등 검증→반영 루프가 돌며, 엄격 트랙은 적대적 검증 + 보안 게이트가 추가됩니다.
※ 모든 문서 산출물은 자족형 HTML로 생성됩니다. docs/Templates/ 템플릿이 있으면 따르고, 없으면 이 문서와 같은 다크 스타일 + SVG 시각화를 사용합니다(베이스: 플러그인 동봉 templates/artifact-base.html).
