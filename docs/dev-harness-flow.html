🧭 dev-harness — 진행 흐름 & 디렉토리 산출/참조 맵

기본판 하네스. 기획 → 설계 → 구현 파이프라인의 각 단계가 어떤 디렉토리를 참조(read)하고 어떤 산출물을 생성(write)하는지 시각화. (강화판은 dev-harness-plus-flow.html 참고)

참조(read) 산출(write) 진행 흐름 선택(필요 시) 기획 설계 구현 마무리

🔎 흐름 단계별 세부 설명

P0컨텍스트 확인

담당: 메인 대화(PM)

  • docs/** 와 기존 산출물을 훑어 초기 실행 / 부분 재실행 / 새 실행을 판별한다.
  • 부분 재실행이면 해당 에이전트만 재호출하고, 새 실행이라도 기존 산출물은 보존(덮어쓰지 않음)한다.
  • 목표·범위·우선순위가 불명확하면 위임 전에 AskUserQuestion으로 먼저 확인한다(추측 금지).
참조docs/** · 기존 산출물산출없음 — 판별 단계

P1기획

담당: product-planner (생성) → reviewer (검증)

  • 기존 자료를 확인한 뒤 PRD·기능명세·사용자 스토리·측정 가능한 수용 기준을 작성한다.
  • 검증·반영 — reviewer가 기획 체크리스트(근거·수용기준·범위·가정/리스크·모순)로 검증 → FAIL이면 반영 항목과 함께 재호출(최대 2회).
  • 통과 조건 — 전체 기능 목록·개발 범위 확정 → 설계로.
참조기존 docs/산출docs/ (PRD·기능명세·스토리) · docs/reviews/

P2아키텍처 설계

담당: chief-architect (생성) → reviewer (검증)

  • docs/(기획)을 입력으로 시스템 아키텍처 + 소프트웨어 아키텍처(추상화) + 디렉토리 구조와 ADR(결정 근거)을 설계한다. 핵심 결정은 대안 2~3개를 비교한다.
  • 검증·반영 — reviewer가 기획 추적성·디렉토리 일관성·근거·과설계 여부를 검증 → 반영.
  • 통과 조건 — 설계 확정 → 개발 인계로.
참조docs/(기획)산출docs/architecture/ (설계서·ADR·API 계약) · docs/reviews/

선택회의 주재 (충돌·합의 필요 시)

담당: 메인 대화(PM) — 필요 시에만

  • 기획과 설계가 충돌하거나(요구 vs 기술 제약) 합의가 필요할 때, 쟁점·각 입장·결정 질문·평가 기준을 정리한다.
  • 관점을 종합해 결론 + 근거를 docs/pm/ 회의록으로 남긴다. 중대한 방향 결정은 AskUserQuestion으로 사용자에게 올린다.
산출docs/pm/ (회의록)

P3개발 인계 (실행 모드 전환)

담당: 메인 대화(PM)

  • 사용자에게 "설계 완료, 개발 시작" 의사를 확인한다.
  • 실행 모드 전환 — 서브에이전트 파이프라인 → 에이전트 팀(team-dev). 세션당 팀은 1개만 활성화한다.
산출없음 — 모드 전환 단계

P4개발 로드맵

담당: tech-lead (생성) → reviewer (검증)

  • 설계의 모듈·의존성을 바탕으로 구현을 순서 있는 단계로 분해해 docs/roadmap.md 를 작성한다.
  • 검증·반영 — reviewer가 설계 정합성·단계 순서·완료 기준을 검증 → 반영.
참조docs/architecture/산출docs/roadmap.md · docs/reviews/

P5단계별 구현 (이중 검증 루프)

담당: tech-lead 주관 — devops · backend · frontend + qa

  • 계획→구현 — 세부 계획을 reviewer가 검증, 통과하면 devops 골격 → backend·frontend가 API 계약 합의 후 병렬 구현한다.
  • 구현 검증 — qa-engineer가 "계획대로 됐는지" 경계면 교차 비교·수용 기준으로 검증하고 버그를 리포트 → 반영.
  • 통과 조건 — DoD 충족 시 tech-lead가 통합하고 다음 단계로. 단계당 최대 2회 반복 후 미해결은 PM에 보고.
참조docs/architecture/ (API 계약·모듈 경계)산출소스 코드 · .env.example · docs/reviews/

P6전체 검증 → 최종 마무리

담당: qa-engineer + reviewer

  • 전체 통합 검증 — qa가 전체 빌드·실행·핵심 시나리오로 통합 정합성을 종합 검증하고, 동시에 reviewer가 기획 수용 기준 전체를 추적성 점검한다.
  • 결함이 있으면 tech-lead가 해당 단계로 회부해 수정 → 재검증.
  • 마무리 — 통과 시 산출물 경로·구현 요약·남은 리스크/후속 제안을 사용자에게 보고하고, 필요 시 팀을 정리(TeamDelete)한다.
참조전체 산출물 · docs/(수용 기준)산출최종 보고 (소스 코드 확정)

📂 디렉토리 지도 (전체)

디렉토리 / 파일구분역할생성·갱신 단계
docs/산출기획 산출물 — PRD·기능명세·사용자 스토리Phase 1
docs/architecture/산출설계 산출물 — 설계서·ADR·API 계약Phase 2 (구현 중 계약 갱신)
docs/roadmap.md산출개발 로드맵 — 순서 있는 구현 단계Phase 4
docs/reviews/산출검증 보고 — reviewer(문서·계획) · qa(동작) 결과모든 검증 단계 (1·2·4·5·6)
docs/pm/산출PM 산출물 — 회의록·실행계획·진행보고회의 주재(필요 시) / 보고
소스 코드 / .env.example산출실제 구현물 — 모듈·API·UI·골격Phase 5
plus와의 차이 — 기본판은 입력 기준 폴더(docs/Planning·Design·Development), 추적성 매트릭스(docs/traceability.md), 코드베이스 감사·리스크 게이팅·적대적 검증·보안 게이트, 그리고 security-reviewer·code-reviewer·tech-writer·debugger 전담 검증/지원 역할이 없습니다. 검증은 reviewer(문서·계획) + qa(동작) 이중 검증으로만 이뤄집니다.
※ 각 화살표(생성/검증/반영) 단위마다 reviewer·qa 검증→반영 루프가 돌며, 단계당 최대 2회 반복 후 미해결은 사용자에게 보고합니다.
