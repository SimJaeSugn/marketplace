# team-dev — 개발팀 소집 및 구현 오케스트레이션

설계가 마무리된 뒤의 **구현 단계** 절차다. 실행 모드는 **에이전트 팀**
(`TeamCreate` + `SendMessage` + 공유 `TaskCreate`) — 개발자들이 서로 직접 소통하며 자체 조율한다.
이 스킬은 (A) 메인 대화(PM)가 팀을 소집·조율하는 절차와, (B) 개발자 에이전트 공통 규칙을 함께 담는다.

> **에이전트 네임스페이스(중요).** 이 스킬은 `dev-harness-plus` 플러그인의 강화 에이전트를 쓴다.
> 원본 dev-harness와 공존할 수 있으므로 팀 소집·호출 시 **`dev-harness-plus:` 접두사**를 붙인다
> (예: `dev-harness-plus:tech-lead`). 아래 표는 가독성을 위해 접두사를 생략한다.

## 선행 조건

- `docs/architecture/`에 설계 문서가 있어야 한다. 없으면 먼저 `dev-orchestrator`로 기획→설계를 끝낸다.
- 설계가 불충분하면 개발을 시작하지 말고 PM이 `chief-architect`에 보완을 요청한다.

## 팀 구성 (역할 기반)

| 에이전트 | 역할 |
|----------|------|
| `tech-lead` | 팀 리더 — 작업 분해·배정, 인터페이스 중재, 통합·리뷰, PM에 보고 |
| `frontend-dev` | UI·클라이언트 로직 구현 |
| `backend-dev` | 서버 로직·API·데이터 계층 구현 |
| `qa-engineer` | 구현 검증 — 점진적 동작·통합 정합성 (경계면 교차 비교) |
| `code-reviewer` | 코드 품질 검증 — 가독성·중복·복잡도·코드 결함 (동작 아님) |
| `security-reviewer` | 보안 검증 — 취약점·인가·시크릿 (보안 게이트) |
| `debugger` | 난항 버그 근본 원인 규명 (원인 불명·간헐적 결함) |
| `devops-engineer` | 프로젝트 골격·빌드·실행 환경·통합 |
| `tech-writer` | 사용자·개발자 문서 (기능 완성 후) |
| `reviewer` | 계획 검증 — 로드맵·단계별 세부계획을 독립 검증 (코드 아님, 계획·문서) |

> 한 역할에 일이 몰리면 그 역할 에이전트를 **병렬로 여러 태스크에 호출**해 처리량을 늘린다(사실상 인원 확장).
> 팀원이 많을수록 조율 비용이 커지므로, 동시에 띄우는 수는 작업 의존성에 맞춰 조절한다.
> **검증자 4종 분업:** `qa-engineer`=동작, `code-reviewer`=코드 품질, `security-reviewer`=보안,
> `reviewer`=계획·문서. 영역이 겹치지 않게 각자에게 자기 관점만 지시한다.

## A. PM(메인 대화)의 오케스트레이션 절차

### Phase 0: 컨텍스트 확인
- `Glob`로 `docs/architecture/`(설계 존재)와 소스 디렉터리(기존 코드 존재)를 확인한다.
- 초기 구현 / 기존 코드에 이어서 / 특정 모듈만 재구현 중 어느 것인지 판별한다. 기존 코드는 보존한다.

### Phase 1: 팀 소집
- `TeamCreate`로 개발팀을 구성한다 (리더: `tech-lead`, 멤버: 위 5종 중 필요한 역할).
- 모든 `Agent`/팀원 호출에 `model: "opus"`.
- 세션당 팀은 1개만 활성화된다. 기획/설계 팀과 겹치지 않게, 설계가 끝난 뒤 이 팀을 만든다.

### Phase 2: 개발 로드맵 (생성→검증→반영)
- `tech-lead`에게 설계 경로(`docs/architecture/`)와 구현 목표·범위를 전달한다.
- `tech-lead`가 설계의 모듈·의존성을 바탕으로 **개발 로드맵**(`docs/roadmap.html`)을 작성한다 — 순서 있는 단계로 분해.
- `reviewer`로 로드맵을 검증하고 반영한다(설계 정합성·단계 순서·완료 기준). PASS 후 Phase 3로.

### Phase 3: 단계별 구현 (각 단계마다 다면 검증 루프)
로드맵의 각 단계를 다음 루프로 진행한다. `tech-lead`가 주관한다.
1. **착수 기준(DoR) 확인**: 그 단계를 시작할 준비가 됐는지 본다(아래 DoR). 미충족이면 먼저 채운다.
2. **세부 계획(코드베이스 수준)**: 그 단계에서 만들 파일·함수·인터페이스·변경 지점을 구체화.
3. **계획 검증 → 반영**: `reviewer`가 설계 정합성·실행가능성 검증 → FAIL이면 반영 후 재검증.
4. **구현 시작**: 통과한 계획대로 팀원 배정. 순서 가이드 — ① `devops-engineer` 골격(첫 단계)
   → ② `backend-dev`·`frontend-dev`가 API 계약 합의 후 병렬 구현 → 팀원은 `SendMessage`로 자체 조율.
5. **다면 검증(병렬) → 반영**: 구현이 끝나면 관점별 검증자를 **병렬로** 돌려 사각을 줄인다.
   - `qa-engineer` — 동작·경계면 교차 비교·수용 기준 (항상)
   - `code-reviewer` — 코드 품질·중복·결함 (코드 변경이 있으면 항상)
   - `security-reviewer` — 보안 게이트 (인증/입력/시크릿/외부 경계를 건드리면 필수)
   각 검증자의 지적을 담당 개발자에 회부해 반영 → 재검증.
6. **난항 버그는 debugger로.** `qa-engineer`가 버그를 찾았으나 원인이 불분명/간헐적이면 추측 수정 대신
   `debugger`에 근본 원인 규명을 맡긴다.
7. 통과하면 `tech-lead`가 통합하고 다음 단계로. 검증은 단계당 **최대 2회** 반복 후 미해결은 PM에 보고.
   **단, 보안 게이트의 치명/높음 항목은 이 한도와 무관하게 미해결 시 다음 단계로 넘기지 않는다.**

### Phase 4: 모니터링 · 종합
- 팀원들은 `SendMessage`로 자체 조율한다. PM은 직접 일하지 않고 진행을 모니터링한다.
- `tech-lead`의 라운드 보고(완료/진행/블로커, 통합 결과)를 받아 사용자에게 종합 보고한다.
- 팀이 못 푸는 결정(범위 변경, 설계 모순)은 PM이 `AskUserQuestion`으로 사용자에게 올린다.

### Phase 5: 전체 검증 → 최종 마무리
모든 로드맵 단계가 끝나면, 단계별 점진 검증과 별개로 **전체 통합 검증**을 한 번 더 거친다 (병렬).
1. **전체 검증(병렬)**:
   - `qa-engineer` — 전체 빌드·실행·핵심 시나리오로 통합 정합성을 종합 검증.
   - `reviewer` — 최종 산출물이 **기획 수용 기준 전체**를 충족하는지 + `docs/traceability.html`에 빈 칸이 없는지 추적성 점검.
   - `security-reviewer` — 보안 게이트 최종 점검(의존성 취약점·시크릿·인가). 고위험 프로젝트면 필수.
   - (선택) `code-reviewer` — 전체 코드 품질 최종 스윕.
2. **반영**: 결함이 있으면 `tech-lead`가 해당 단계로 회부해 수정 → 재검증.
3. **문서화(필요 시)**: 사용자/개발자가 쓸 문서가 필요하면 `tech-writer`로 README·사용 가이드를 코드와 동기화해 작성한다.
4. **최종 마무리**: 전체 검증 통과 시 산출물 경로 + 구현 요약 + 추적성 매트릭스 상태 + (기존 코드 변경 시) `docs/pm/change-log.html` 이력 요약 + 남은 리스크/후속 제안을 사용자에게 보고한다.
5. 필요 시 팀을 정리(`TeamDelete`)하고, 다음 작업(추가 기능, 배포 등)을 제안한다.

## B. 공유 개발 규칙 (개발자 에이전트 공통)

### 파트 기준 문서 (개발 파트 공통)
- 모든 개발자·검증자는 착수 전에 **`docs/Development/`** 폴더를 확인한다. 있으면 그 안의 코딩 표준·룰·
  가이드를 **준수 기준**으로 삼고, 완료 판정(DoD)에 룰 준수 여부를 포함한다.
- `frontend-dev`는 **`docs/Design/`**(디자인 시스템·UI 가이드)도 함께 기준으로 삼는다.
- 해당 폴더가 **없으면** 외부 기준 없이 주변 코드 스타일·모범 사례로 스스로 진행한다(중단하지 않는다).
- `tech-lead`는 태스크 배정 시 이 기준 폴더 경로를 팀원에게 명시한다.

### 코딩 표준
- **기존 코드에 맞춘다.** 새 컨벤션을 들이지 말고, 주변 코드의 스타일·네이밍·구조를 따른다.
- 설계(`docs/architecture/`)의 모듈 경계·계약을 준수한다. 임의로 경계를 넘지 않는다.
- 비밀정보(키·토큰)는 커밋하지 않는다. 환경변수는 예시 파일로 관리한다.

### 인터페이스 규칙
- 프론트·백엔드 경계의 API 계약(엔드포인트·요청/응답 shape·에러 형식)은 **추측하지 않고 합의**한다.
- 계약이 정해지거나 바뀌면 즉시 상대에게 `SendMessage`로 통지하고 약속된 경로에 기록한다.

### 착수 기준 (Definition of Ready)
한 태스크는 다음이 갖춰져야 **시작**할 수 있다 (없으면 먼저 채운다):
1. 무엇을/왜 만드는지 명확하다 (어떤 설계 요소·요구를 구현하는가)
2. 의존하는 인터페이스·API 계약이 합의돼 있다 (추측 착수 금지)
3. 완료 기준(아래 DoD 중 해당 항목)이 정의돼 있다
4. 필요한 골격·환경이 준비돼 있다 (없으면 `devops-engineer` 선행)

### 완료 기준 (Definition of Done)
한 태스크는 다음을 모두 만족해야 완료다:
1. 빌드/타입체크/린트 통과 (가능한 경우 `Bash`로 실제 실행해 확인)
2. 설계 계약·기획 수용 기준 충족, **추적성 매트릭스의 해당 행 갱신**
3. 동작 검증 또는 테스트 존재 (핵심 경로는 테스트로 보장)
4. 경계면(의존하는 다른 모듈)과 shape이 일치
5. **보안 점검 통과** — 입력 검증·인가·시크릿 비노출 (보안 관련 변경 시 `security-reviewer` 게이트)
6. **코드 품질 통과** — `code-reviewer`의 정확성 결함 0건
7. `tech-lead` 리뷰 통과

### 변경 이력 기록 (기존 코드가 있을 때)
- 기존 코드/산출물이 있는 상태에서 기능을 **추가·수정·삭제**하면, 그 변경이 확정된 직후
  **`docs/pm/change-log.html`** 에 이력을 기록한다(최신이 위로). 형식: 날짜·회차·유형·대상(기능/요구 ID)·내용·사유·영향 산출물.
- **삭제는 사유를 반드시 남긴다.** 코드를 무단 삭제하지 말고 결정과 근거를 이력에 남긴다.
- 변경된 요구 ID는 `docs/traceability.html` 행 상태도 함께 갱신한다. 기록은 `tech-lead`가 하거나 `tech-writer`에 위임한다.

### 소통 프로토콜 (요약)
- 진행/완료/블로커는 `tech-lead`에게 보고하고 `TaskUpdate`로 상태를 갱신한다.
- 다른 역할에 필요한 게 있으면 직접 `SendMessage`한다 (예: 프론트→백엔드 API 질의).
- 코드 전문을 메시지에 복붙하지 말고 파일 경로 + 요약으로 소통한다.

## 에러 핸들링
- 태스크가 완료 기준 미달이면 `tech-lead`가 1회 재작업 지시 후 재배정. 재실패 시 블로커로 PM에 보고.
- 통합 빌드 실패 시 `devops-engineer`가 원인을 좁혀 해당 모듈 담당에게 회부한다.
- 설계와 구현이 충돌하면 임의로 코드를 비틀지 말고 `tech-lead` → PM → (필요시) `chief-architect` 경로로 올린다.

## 테스트 시나리오
**정상 흐름:** PM "설계 끝났으니 개발 시작" → 팀 소집 → devops 골격 → 백/프론트 API 계약 합의 →
병렬 구현 → 모듈별 QA 점진 검증 → tech-lead 통합 → PM이 사용자에게 보고.
**에러 흐름:** 프론트가 기대한 응답 shape과 백엔드 실제 응답 불일치 → QA가 경계면 교차 비교로 발견 →
backend-dev에 버그 리포트 → 수정 후 재검증 → 통합.
