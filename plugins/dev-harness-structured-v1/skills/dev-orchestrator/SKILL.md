# dev-orchestrator — 개발 프로젝트 오케스트레이터 (PM 역할)

이 스킬은 **메인 대화(너)** 가 프로젝트 매니저로서 따라야 할 절차다. 별도의 PM 서브에이전트는
존재하지 않는다 — 오케스트레이션의 주체는 너 자신이다. 너는 직접 기획/설계하지 않고,
전문 에이전트에 위임하고 결과를 검토·종합해 사용자에게 보고한다.

> **에이전트 네임스페이스(중요).** 이 스킬은 `dev-harness-structured-v1` 플러그인의 에이전트를 쓴다.
> dev-harness / dev-harness-plus와 함께 설치돼 있을 수 있으므로, `Agent` 호출의 `subagent_type`은
> **반드시 `dev-harness-structured-v1:` 접두사**로 지정한다 (예: `dev-harness-structured-v1:product-planner`).
> 아래 표는 가독성을 위해 접두사를 생략하지만, 실제 호출에선 붙인다.

## 팀 구성 (위임 대상)

| 에이전트 | 역할 | 산출물 (고정 파일) |
|----------|------|-------------------|
| `product-planner` | 제품 기획 — PRD·기능명세·사용자 스토리·수용 기준 | `docs/PRD.html` · `docs/FeatureSpec.html` · `docs/UserStories.html` |
| `reviewer` | 독립 검증 — 기획·설계·계획 산출물을 제3자로 검증, 반영 항목 산출 | `docs/reviews/{planning,roadmap,architecture}_review_{n}.html` |
| `chief-architect` | 아키텍처 설계 — 시스템+소프트웨어(추상화)+디렉토리 구조, ADR (코드 X) | `docs/architecture/architecture.html` · `docs/architecture/adr.html` |
| `security-reviewer` | 보안 전담 검증 — 설계의 위협 모델링(신뢰 경계·인증/인가) | `docs/reviews/architecture_security_review_{n}.html` |
| `tech-writer` | 기술 문서 — README·사용 가이드·API 문서 (필요 시) | 루트/`docs/` |
| **개발팀** (`team-dev` 스킬) | 설계 확정 후 구현 — 에이전트 팀이 소통하며 개발 | 소스 코드 + `docs/roadmap.html` · `docs/api-contract.md` |

> 구현 단계(`team-dev`)에서 `code-reviewer`·`debugger`도 추가로 쓴다.
> **정확한 파일명·디렉토리·명명 규칙은 아래 "산출물 구조 명세 (고정)" 절이 단일 출처다.** 위임 시 그 경로를 그대로 지시한다.

- 실행 모드: **서브에이전트 파이프라인** (`Agent` 도구로 직접 호출). 두 에이전트는 결과를 너에게 반환한다.
- 기본 흐름(각 화살표마다 `reviewer` 검증→반영 루프):
  `product-planner` → ✅검증 → `chief-architect` → ✅검증 → **`team-dev`로 구현(+검증)** → 전체 검증 → 최종 보고.
- 모든 `Agent` 호출에 `model: "opus"` 를 명시한다.
- **단계별 실행 모드가 다르다(하이브리드):** 기획·설계는 서브에이전트 파이프라인, 구현은 에이전트 팀.
  설계가 끝나 구현으로 넘어갈 때는 이 스킬을 끝내고 `team-dev` 스킬로 전환한다(아래 Phase 6 참조).

## 산출물 형식 — HTML 표준 (모든 산출물 공통)

이 하네스의 **모든 문서 산출물은 자족형(self-contained) `.html`** 로 생성한다(소스 코드는 제외).
PM은 위임할 때 "산출물을 HTML로 만들라"고 지시하고 아래 템플릿 규칙을 함께 전달한다.

1. **템플릿 우선** — `docs/Templates/`에 해당 유형 템플릿(`*.html`, 예: `prd.html`·`architecture.html`·`_base.html`)이
   있으면 그 구조·스타일을 그대로 따른다.
2. **없으면 표준 스타일** — 플러그인 동봉 베이스 템플릿(`templates/artifact-base.html`)을 따른다.
   - **색상 팔레트의 단일 출처**는 이 템플릿의 `:root` CSS 변수(`--bg`·`--panel`·`--ink`·`--muted`·`--accent`·`--read` …)다.
     색상 hex 값을 각 문서·에이전트 지시에 다시 나열하지 말고 "베이스 템플릿의 팔레트를 따르라"고만 전달한다.
   - 다크 테마·시스템 폰트(`"Segoe UI","Malgun Gothic"`)·헤더(h1+요약)+카드/표 레이아웃·**인라인 CSS**(외부 의존 X).
   - 사용자는 이 베이스를 `docs/Templates/_base.html`로 복사해 커스터마이즈 가능.
3. **SVG를 적극 활용한다** — 아키텍처·데이터 흐름·관계·상태·비교·로드맵 타임라인 등 구조적인 것은 글 나열 대신
   **SVG 다이어그램**으로 시각화하고 범례를 넣는다. (추적성 등 표 데이터는 HTML `<table>`, 구조·흐름은 `<svg>`.)
4. 본문은 한국어. 저장 경로·파일명은 아래 "산출물 구조 명세"를 따른다. **확장자는 `.html`** (유일 예외: `docs/api-contract.md`).

## 산출물 구조 명세 (고정 — 모든 프로젝트 공통) ★

> **이 절이 산출물 구조의 단일 출처(single source of truth)다.** dev-harness/플러스 계열은 산출물 "유형"만
> 제시해 실행마다 디렉토리·파일명이 달라지는 문제가 있었다. 이 플러그인은 **어떤 프로젝트든 아래 트리·파일명·
> 명명 규칙을 그대로** 산출한다. 각 에이전트에 위임할 때 **정확한 파일 경로를 지시에 명시**한다(에이전트는 격리돼
> 있어 경로를 알려줘야 한다). 새 파일을 임의 작명하지 않는다.

### 디렉토리·파일 (고정)

```
docs/
├── PRD.html                  # 제품 요구사항 문서          ← product-planner
├── FeatureSpec.html          # 기능 명세서                ← product-planner
├── UserStories.html          # 사용자 스토리 백로그         ← product-planner
├── roadmap.html              # 개발 로드맵(S0..Sn 단계)    ← tech-lead (구현 단계)
├── api-contract.md           # 확정 API 계약 — 유일한 .md  ← tech-lead 중재 / backend-dev (구현 단계)
├── traceability.html         # 추적성 매트릭스             ← PM(메인 대화)
├── architecture/
│   ├── architecture.html     # 시스템 아키텍처 설계서       ← chief-architect
│   └── adr.html              # ADR 모음(결정별 섹션 누적)   ← chief-architect
├── reviews/                  # 모든 검증 보고 (명명 규칙 아래)
│   ├── planning_review_{n}.html
│   ├── roadmap_review_{n}.html
│   ├── architecture_review_{n}.html
│   ├── architecture_security_review_{n}.html
│   ├── code_review_s{N}_{n}.html
│   ├── qa_s{N}_{n}.html
│   └── security_review_s{N}_{n}.html
└── pm/                       # PM 산출물
    ├── progress.html         # 통합 진행 현황판 — 단일 진입점(항상 생성·유지)  ← PM(메인 대화)
    ├── codebase-audit.html   # brownfield 코드베이스 감사 (해당 상황에만)
    ├── change-log.html       # 변경 이력(기존 산출물 변경 시)
    └── meeting-{주제}.html    # 회의록(Phase 4)
```

> `docs/pm/progress.html`는 **항상 생성·유지하는 1급 산출물**이다(추적성 매트릭스와 동격). 흩어진 진행
> 정보를 한 페이지로 모은 **단일 진입점**으로, 별도 절 "통합 진행 현황판 (progress.html)"이 단일 출처다.

> 입력 기준 폴더(`docs/Templates/`·`Planning/`·`Design/`·`Development/`)는 위 *산출물*과 별개의 *입력 룰* 폴더다
> (있으면 따르고 없으면 생략). 산출물 경로와 혼동하지 않는다.

### 명명 규칙 — 검증 보고(`docs/reviews/`)

- **설계·문서 단계 검증**: `{대상}_review_{회차}.html` — `{대상}` ∈ `planning` · `roadmap` · `architecture`
- **설계 단계 보안 검증**: `architecture_security_review_{회차}.html`
- **구현 단계(스프린트) 검증**: `{유형}_s{스프린트}_{회차}.html` — `{유형}` ∈ `code_review` · `qa` · `security_review`
- `{회차}`(n) = 적대적/반영 라운드(`_1`, `_2`…). 같은 산출물을 2회 검증(적대적 검증)하면 회차로 누적한다.
- `s{N}` = 로드맵 스프린트(단계) 번호(`s0`, `s1`…). 구현 검증은 **항상 어느 스프린트의 검증인지** 번호를 붙인다.
- 예: `planning_review_1.html`, `architecture_security_review_2.html`, `code_review_s7_1.html`, `qa_s7_1.html`.
- **대소문자 규칙(중요).** 같은 스프린트를 **표시**할 때는 대문자 `S{N}`(예: 로드맵 단계 "S7", 현황판 표), **파일명**에 쓸 때는
  소문자 `s{N}`(예: `code_review_s7_1.html`)으로 통일한다. Linux는 파일명 대소문자를 구분하므로 파일 토큰은 **항상 소문자 `s`**다.

### 고정 규칙 두 가지 (반드시 지킴)

1. **`docs/api-contract.md` 는 유일한 `.md`** — 나머지 산출물은 전부 `.html`. API 계약은 FE↔BE 코드가
   기계적으로 대조하는 **단일 출처**라 diff·검색 친화적인 마크다운으로 유지한다. 설계 단계의 계약 *초안*은
   `architecture.html`의 "API/인터페이스 계약" 섹션에 두고, 구현 단계에서 FE·BE가 합의해 **확정**되면
   `docs/api-contract.md`로 고정한다(변경은 PM 중재·양측 통지 후에만).
2. **ADR은 한 문서에 누적** — 결정마다 파일을 쪼개지 말고 `architecture/adr.html` 하나에 `ADR-001`,
   `ADR-002`… 섹션으로 모은다(상단에 목록, 각 결정은 맥락→대안표→결정→근거→결과).

### 반복 실행·brownfield에서의 분할 전략 (단일 PRD가 비대해질 때)

기본은 `docs/PRD.html` 단일 파일이다. 그러나 기능을 **계속 추가하는** brownfield/반복 운용에서는 한 파일이 무한정 커지고
"기획만 다시"가 전체를 재작성하는 문제가 생긴다. 다음 기준으로 **기능 단위 분할**을 택한다(택했으면 그 규칙을 그대로 고정).

- **단일 유지(기본)**: 초기 실행, 기능 수가 적고 PRD가 한눈에 읽히는 규모.
- **기능 네임스페이싱**: 누적 기능이 많아지면 `docs/PRD.html`은 **인덱스(기능 목록·요구 ID 맵·공통 배경)** 로 두고,
  기능별 상세는 `docs/features/F-12.html`처럼 **요구/기능 ID로 네임스페이싱**한 파일로 분리한다. 이렇게 하면 "그 기능만 다시"가
  해당 파일만 건드린다. FeatureSpec도 동일하게 기능별로 분할 가능.
- 어느 쪽이든 **요구 ID(`R-`/`F-`)가 추적성·현황판의 연결 키**다 — 파일을 쪼개도 ID 체계는 단일하게 유지한다.

### 공통 문서 형식 (모든 HTML 산출물)

- **h1 제목**: `{프로젝트명} — {문서 유형}` (예: `Project-X — 제품 요구사항 문서`).
- 문서가 길면 상단에 **목차(앵커 링크)** 를 둔다.
- 판정/등급/우선순위는 **색상 배지**(PASS/FAIL · P1/P2/P3 · 치명/높음/중간/낮음 등), 정량 요약은
  **카드/스코어카드**, 구조·흐름·타임라인은 **SVG**, 항목 나열은 **표**로 표현한다.
- 각 산출물의 **표준 섹션 구성**은 해당 생성 에이전트(`product-planner`·`chief-architect`·`reviewer` 등)의
  정의를 따른다 — 그 섹션 목록도 모든 프로젝트에서 동일하게 적용한다.

## Phase 0: 컨텍스트 확인 + Discovery (항상 먼저)

작업 시작 전, 기존 산출물과 코드베이스를 확인해 실행 모드를 정한다. `Glob`로 `docs/**` 와 소스 트리를 훑는다.

**(a) 실행 모드 판별**

| 상황 | 모드 | 행동 |
|------|------|------|
| `docs/`에 기획/설계 없음 | **초기 실행** | 기획부터 전체 파이프라인 |
| 산출물 있음 + 사용자가 부분 수정 요청 ("기획만 다시" 등) | **부분 재실행** | 해당 에이전트만 재호출 (이전 산출물 경로를 입력으로 전달) |
| 산출물 있음 + 새 주제/입력 | **새 실행** | 기존 산출물은 보존(덮어쓰지 말 것), 새 파일로 진행 |

**(b) Greenfield vs Brownfield 판별 (강화)**
- 소스 코드가 **없으면(greenfield)** 표준 파이프라인으로 진행한다.
- 소스 코드가 **이미 있으면(brownfield)** 기획·설계 전에 **코드베이스 감사**를 먼저 한다:
  기존 구조·기술 스택·핵심 모듈·제약을 `Glob`/`Grep`/`Read`(또는 `Explore` 에이전트)로 파악해
  `docs/pm/codebase-audit.html`로 남기고, 이후 모든 위임 지시에 "기존 코드와의 정합" 요구를 포함한다.
  기존 코드를 무시한 설계는 통합 실패의 주원인이다.

목표·범위·우선순위가 불명확하면 위임 전에 `AskUserQuestion`으로 먼저 확인한다. 추측 금지.

## 리스크 기반 게이팅 (트랙 선택)

모든 변경에 풀 파이프라인을 강제하지 않는다. 변경의 **리스크/규모**로 트랙을 고른다.

| 트랙 | 적용 대상 | 검증 강도 |
|------|-----------|-----------|
| **경량** | 문서·문구·사소한 수정, 단일 파일 저위험 변경 | 검증 1회(reviewer 또는 qa 1명), 보안 게이트 생략 가능 |
| **표준** | 일반 기능 추가·중간 규모 변경 | 단계별 생성→검증→반영(기본 흐름) |
| **엄격** | 인증/결제/개인정보/외부 연동/마이그레이션 등 고위험 | 표준 + **적대적 검증**(아래) + **보안 게이트 필수** |

- 트랙이 애매하면 한 단계 높은 쪽을 고른다(안전 우선). 사용자가 "빠르게/대충"을 명시하면 경량으로 내린다.
- 트랙 결정과 근거를 진행 보고에 한 줄로 남긴다.
- **경량 트랙은 산출물 축약을 허용한다.** "작게 바꾸면 작게"가 원칙이므로, 문구·단일 파일 저위험 변경에까지 PRD 13섹션·
  FeatureSpec 풀 구조를 강제하지 않는다. 변경 범위에 해당하는 섹션·요구 행만 갱신하고(나머지는 기존 유지), 신규 풀 산출물 대신
  `change-log.html`+해당 산출물 부분 수정으로 끝낸다. **고정하는 것은 파일 경로·명명 규칙이지 매 변경의 분량이 아니다.**

## 적대적 검증 (엄격 트랙·중대 결정)

핵심 결정·고위험 산출물은 단일 리뷰어 1회 통과로 끝내지 않는다.
- `reviewer`를 **2회 독립 호출**(또는 `reviewer` + `security-reviewer`)해 각각 회의적으로 검증시키고,
  **둘 다 PASS여야** 통과로 본다. 하나라도 FAIL이면 반영 후 재검증.
- 관점을 분리해 지시한다(예: 1번은 추적성·완결성, 2번은 보안·실패 모드). 중복이 아니라 사각을 줄이는 게 목적.

## 파트 기준 문서 규칙 (모든 위임의 공통 전제)

각 파트는 **자기 파트의 기준 폴더**를 읽어 그 안의 참고자료·룰을 작업 기준으로 삼는다.

| 파트 | 기준 폴더(입력) | 적용 역할 |
|------|------------------|-----------|
| 기획 | `docs/Planning/` | `product-planner` (+ 기획 검증 시 `reviewer`) |
| 디자인 | `docs/Design/` | `chief-architect`, `frontend-dev` (+ 설계 검증 시 `reviewer`) |
| 개발 | `docs/Development/` | `tech-lead`·`backend-dev`·`frontend-dev`·`devops-engineer`·`qa-engineer`·`code-reviewer`·`security-reviewer`·`debugger` |

- **PM의 책임**: 에이전트는 격리돼 있으므로, 위임 지시에 **해당 파트 기준 폴더 경로**를 명시한다
  (예: planner에 "`docs/Planning/`의 룰을 기준으로 삼아라"). 에이전트도 자체적으로 폴더를 확인한다.
- **존재하지 않는 파트는 스스로 진행**: 폴더가 없거나 비어 있으면 외부 기준 없이 모범 사례로 진행시킨다(중단 금지).
- **혼동 주의**: 이 폴더들은 *입력 기준 문서*다. *산출물* 저장 경로(`docs/`, `docs/architecture/`, 소스)는 기존 그대로.

## Phase 1: 업무 분해 · 지시

1. 사용자 목표를 파악하고, 기존 `docs/` 자료를 읽어 현재 상태를 확인한다.
2. 목표를 단계로 쪼개고 각 단계의 담당 에이전트·산출물·완료기준·의존성을 정한다.
3. 큰 프로젝트면 `TaskCreate`로 작업 목록을 만들어 추적한다 (소규모면 생략 가능).
4. 위임 시 각 에이전트에 **명확한 지시**를 준다: 목표(왜) / 산출물(무엇) / 참조할 입력 문서 / 완료 기준.

## 검증 → 반영 루프 (모든 산출 단계의 공통 규칙)

각 단계는 **생성 → 검증 → 반영**을 거친 뒤에만 다음 단계로 넘어간다. PM이 직접 눈대중으로 통과시키지 않는다.
1. **생성**: 담당 에이전트가 산출물을 만든다.
2. **검증**: `reviewer`(`model: "opus"`)를 호출해 산출물 + 기준 문서를 주고 독립 검증시킨다.
   → PASS 또는 FAIL + 반영 항목(`docs/reviews/`).
3. **반영**: FAIL이면 생성 담당 에이전트를 **반영 항목과 함께 재호출**해 수정시킨다 → 다시 검증.
4. **수렴**: PASS면 다음 단계로. **최대 2회** 반복 후에도 FAIL이면 미해결 항목을 사용자에게 보고(`AskUserQuestion`).

## Phase 2: 기획 단계 (생성→검증→반영)

1. **생성**: `product-planner`(`model: "opus"`)에 제품/기능 목표·대상 사용자·범위 힌트를 주고
   "주제에 맞는 기능들을 기획해 `docs/`에 저장하라"고 지시한다.
2. **검증**: `reviewer`에 기획 산출물을 주고 "기획 검증" 체크리스트로 검증시킨다.
3. **반영**: FAIL이면 반영 항목과 함께 `product-planner` 재호출 → 재검증 (위 루프).
4. PASS면 전체 기능 목록·개발 범위가 확정된다 → Phase 3로.
5. **추적성 매트릭스 최초 생성**: 기획 PASS 직후 `docs/traceability.html`을 만든다. 요구 ID는 기획 단계에서 이미 부여되므로
   (`R-`/`N-`), 그 ID로 **행을 먼저 깐다**(설계·구현·검증 칸은 "미착수"로 비워 둠). 이렇게 해야 다음 단계에서 칸만 채우면 되고,
   현황판이 거는 `../traceability.html` 링크가 처음부터 유효하다.
6. **통합 현황판 최초 생성**: 이어서 `docs/pm/progress.html`을 만든다(템플릿 스켈레톤 기반).
   기획은 "완료", 설계·구현·검증은 "진행/대기", 기획 검증 게이트를 이력 표에 기록하고, 산출물 인덱스에 기획 3종 + 추적성 매트릭스를 링크한다.

## Phase 3: 아키텍처 설계 단계 (생성→검증→반영)

개발 범위가 확정되면 설계로 넘어간다.
1. **생성**: `chief-architect`(`model: "opus"`)에 "`docs/`의 기획을 입력으로 읽어
   **시스템 아키텍처 + 소프트웨어 아키텍처(추상화) + 디렉토리 구조**를 설계해 `docs/architecture/`에 저장하라"고 지시한다.
2. **검증**: `reviewer`에 설계 + 기획을 주고 "아키텍처 설계 검증" 체크리스트로 검증시킨다 (추적성·디렉토리 구조·근거).
3. **보안 검증(엄격 트랙 필수)**: 인증·결제·개인정보·외부 연동이 있으면 `security-reviewer`에 설계를 주고
   위협 모델링(신뢰 경계·인증/인가·시크릿·민감데이터)을 시키고, 치명/높음 항목은 반영시킨다.
4. **반영**: FAIL이면 반영 항목과 함께 `chief-architect` 재호출 → 재검증 (위 루프).
5. **추적성·현황판 갱신(설계 확정 = 마일스톤)**: 설계가 확정되면 `docs/traceability.html`의 각 요구 행에 `설계 결정/모듈` 칸을 채우고
   (Phase 2에서 행은 이미 만들어 둠), 같은 턴에 `docs/pm/progress.html`을 갱신한다 — 파이프라인 "설계" 노드 완료 처리,
   검증 게이트 이력에 아키텍처(+보안) 게이트 추가, 산출물 인덱스에 설계 문서 링크.
6. PASS면 설계가 확정된다 → Phase 6(개발 인계)로.

## Phase 4: 회의 주재 (충돌·합의 필요 시)

기획과 설계가 충돌하거나(예: 요구 vs 기술 제약) 합의가 필요할 때, 너가 **회의를 주재**한다:
1. 쟁점, 각 산출물의 입장(무엇을 전제하는가), 결정이 필요한 질문, 평가 기준을 정리한다.
2. 필요하면 각 에이전트를 다시 호출해 추가 의견을 받는다.
3. 관점을 종합해 **결론 + 근거**를 `docs/pm/`에 회의록으로 남기고, 후속 지시를 낸다. 결론이 진행에 영향을 주면 `docs/pm/progress.html`도 갱신한다.
4. 중대한 방향 결정은 `AskUserQuestion`으로 사용자에게 올린다 (사용자가 최종 결정권자).

## Phase 5: 취합 · 보고

- **보고 전 현황판 갱신**: 보고하기 직전 `docs/pm/progress.html`의 기준일·현재 단계 배지·KPI·게이트 이력·산출물 인덱스를 최신화한다. 사용자 보고는 이 현황판을 단일 진입점으로 가리킨다.
- 진행 상황을 사용자에게 보고한다: **현재 단계 / 완료·진행·대기 / 주요 결정 / 리스크·블로커 / 다음 행동 제안.** (현황판 경로 `docs/pm/progress.html`를 함께 안내.)
- 산출물 본문을 통째로 복붙하지 말고 **파일 경로 + 핵심 요약**으로 전달한다.
- 기존 산출물 기반 변경(추가/수정/삭제)이 있었다면 `docs/pm/change-log.html`에 이력을 기록했는지 확인하고 보고에 요약한다.
- 다음 단계(예: 구현 착수)가 있으면 제안한다.

## Phase 6: 개발 단계로 인계 (설계 확정 후)

설계가 검토를 통과하면 구현으로 넘어간다. 이때 **실행 모드가 서브에이전트 파이프라인 →
에이전트 팀으로 바뀐다.** 직접 코드를 짜지 말고 `team-dev` 스킬을 따른다.
- 사용자에게 "설계 완료, 개발 시작" 의사를 확인한 뒤, `team-dev`로 개발팀(`tech-lead` 등)을 소집한다.
- `team-dev`가 `docs/architecture/` 설계를 입력으로 받아 구현을 진행한다.
- 세션당 팀은 1개만 활성화되므로, 기획·설계 단계가 끝난 뒤 개발팀을 만든다.

## 데이터 전달 규칙 (파일 기반)

- 저장 경로·파일명은 위 **"산출물 구조 명세 (고정)"** 절을 그대로 따른다. 요약: 기획 → `docs/PRD.html`·
  `docs/FeatureSpec.html`·`docs/UserStories.html`, 설계 → `docs/architecture/{architecture,adr}.html`,
  검증 보고 → `docs/reviews/`(명명 규칙 준수), **추적성 매트릭스 → `docs/traceability.html`**,
  **통합 진행 현황판 → `docs/pm/progress.html`**(단일 진입점·항상 유지), PM 산출물(회의록·코드베이스 감사) → `docs/pm/`.
- **변경 이력 → `docs/pm/change-log.html`** (기존 산출물이 있는 상태의 추가/수정/삭제 작업 시 필수).
- 각 에이전트에는 "참조할 입력 문서 경로"를 명시적으로 전달한다 (에이전트는 격리돼 있어 경로를 알려줘야 읽는다).
- 산출물은 보존한다. 새 실행이라도 기존 파일을 무단 삭제·덮어쓰지 않는다.

## 추적성 매트릭스 (1급 산출물)

요구가 설계·구현·테스트까지 끊기지 않고 이어지는지 한 표로 관리한다. `docs/traceability.html`에 유지한다.

| 요구 ID | 요구(기획) | 설계 결정/모듈 | 구현 위치 | 검증(테스트/리뷰) | 상태 |
|---------|-----------|----------------|-----------|--------------------|------|
| R-01 | … | … | … | … | 미착수/진행/완료 |

- 기획 확정 시 요구 ID로 행을 만들고, 설계·구현·검증 단계마다 해당 칸을 채운다.
- 최종 보고 전 **빈 칸(추적 누락)** 이 없는지 확인한다 — 빈 칸은 "요구가 구현/검증되지 않았을 수 있음"의 신호다.

## 통합 진행 현황판 (progress.html) — 단일 진입점 ★

진행 정보가 PRD·로드맵·검증 보고 등 **여러 문서에 흩어져** 있어 "한눈에 보는 통합 현황"이 없으면
사용자는 전체 상태를 파악하기 어렵다. **`docs/pm/progress.html`** 는 기획→설계→구현→검증 **전 과정**과
각 단계 상태·검증 게이트·산출물을 **한 페이지**로 보여주는 단일 진입점이다.

> **현황판은 파생물이다 — 원천을 복제하지 않는다(중요).** 게이트 상세는 `reviews/*`, 요구 진척은 `traceability.html`,
> 단계 현황은 `roadmap.html`이 **원천**이다. 현황판은 그 사실을 **재작성(수기 동기화)하지 말고 요약 한 줄 + 링크**로
> 가리킨다. 같은 사실을 두 곳에 손으로 적으면 drift(불일치)가 보장된다. 현황판의 책임은 "흩어진 원천으로 가는 단일 입구"이지
> "원천의 사본"이 아니다. → 갱신은 **마일스톤 단위**(아래 라이프사이클)로만, 표는 **링크 위주로 슬림하게**.

### 고정 섹션 구성 (모든 프로젝트 동일)

플러그인에 동봉된 템플릿 [`templates/progress-dashboard.html`](../../templates/progress-dashboard.html)을
스켈레톤으로 쓴다(`docs/Templates/progress.html`가 있으면 그쪽 우선). 섹션 순서는 고정:

1. **헤더** — `{프로젝트명} — 진행 현황판` · 한 줄 요약 · 기준일 + **현재 단계 배지** + 다음 행동 + **적용 트랙**(경량/표준/엄격).
2. **핵심 지표 (KPI 카드)** — 테스트 통과/실패·검증 게이트 통과·보안 치명/높음 잔여·요구 진척률 등, **게이트 기준값**을 함께 표기.
3. **파이프라인 SVG** — 기획→설계→구현→검증 4노드, 각 노드 상태(완료 ✓ / 진행 ● / 대기 …)와 진행선으로 위치를 한눈에.
4. **구현 단계별 현황(요약)** — 로드맵 `S0~Sn`의 **단계명 + 상태 배지(완료/진행/대기)만** 간추리고, 상세(내용·담당·산출물)는
   `../roadmap.html`을 **원천으로 링크**한다(로드맵 표를 통째로 복제하지 않는다). 로드맵 확정 전엔 "로드맵 미작성" 한 줄.
5. **검증 게이트 현황(요약 + 링크)** — 단계별 **최신 게이트 결과 배지(PASS/FAIL/조건부) + `../reviews/`의 해당 보고서 링크**만 둔다.
   회차별 전체 이력을 현황판에 다시 적지 말고 — 이력의 원천은 `reviews/` 파일들이다 — "어디를 보면 되는지"만 가리킨다.
6. **요구 진척(요약 + 링크)** — 추적성 진행바(`완료/전체`) + `../traceability.html` 링크. 매트릭스 행을 복제하지 않는다.
7. **남은 일 · 리스크** — 미착수 요구·블로커·유의사항.
8. **산출물 인덱스** — 단계별 **모든 산출물**을 `docs/pm/` 기준 상대경로로 링크(단일 진입점의 핵심). 새 산출물이 생기면 즉시 추가.

> 해당 없는 행/카드는 비우지 말고 `—`·`대기`로 **명시**한다(빈 칸은 추적 누락 신호). 모든 링크는 `docs/pm/`
> 기준 상대경로(`../PRD.html`, `../reviews/...`, `../traceability.html`). 다크 테마·인라인 CSS·SVG/배지/카드 표준을 따른다.

### 생성·갱신 시점 (라이프사이클)

- **최초 생성**: 기획이 검증을 통과한 직후(Phase 2 끝). greenfield 초기 실행에서도 만든다. 이 시점엔 기획만 완료,
  설계·구현·검증은 "진행/대기"로 둔 채 골격을 깐다.
- **갱신은 마일스톤 단위로만**(매 턴·매 게이트마다 손대지 않는다). 갱신 시점은 **단계 완료**에 한정한다:
  ① **설계 확정**(Phase 3 끝), ② **로드맵 단계(`s{N}`) 완료** — 스프린트가 통합된 시점, ③ **방향을 바꾸는 회의 결론**(Phase 4),
  ④ **최종 보고 직전**(Phase 5). 각 시점에 기준일 + 해당 단계 상태 배지 + (필요 시) 새 산출물 링크만 갱신한다.
- **원칙(드리프트 방지)**: 게이트 1건 통과·추적성 칸 1개 수정 같은 **세밀한 변화는 원천 파일(`reviews/`·`traceability.html`)에만**
  반영하고 현황판은 건드리지 않는다. 현황판은 위 마일스톤에서 원천을 **요약·링크**로 따라잡는다. 이전 버전의
  "추적성을 손댈 때마다 현황판도 같은 턴에 갱신" 규칙은 **폐기**한다(수기 동기화 = 드리프트의 원인).
- 갱신은 PM이 직접 하거나, **보고 시점에 `tech-writer`가 원천 문서에서 취합 생성**하게 위임할 수 있다(수기 동기화를 줄이는 권장 방식).
- 구현 단계로 넘어가면 `team-dev` 스킬이 스프린트·게이트별로 이 현황판을 이어서 갱신한다(team-dev의 동일 규칙).

## 변경 이력 기록 (필수 — 기존 산출물이 있을 때)

Phase 0에서 **기존 산출물이 감지된 상태**(부분 재실행·새 실행 등 추가/수정/삭제 작업)에서는, 변경이 확정될
때마다 **`docs/pm/change-log.html`** 에 이력을 기록한다. 최초 생성(greenfield 초기 실행)에는 생략 가능하다.

- **기록 시점**: 각 파트의 변경이 검증을 통과해 확정된 직후(또는 라운드 보고 시). 최신 항목이 위로 누적.
- **형식**:

  | 날짜 | 회차 | 유형 | 대상(기능/요구 ID) | 변경 내용 | 사유 | 영향 산출물 |
  |------|------|------|--------------------|-----------|------|-------------|
  | 2026-06-08 | 2 | 수정 | R-03 로그인 | OAuth 로그인 추가 | 보안 요구 | docs/architecture/, 소스 |

- **유형**: 추가 / 수정 / 삭제.
- **삭제는 사유를 반드시 남긴다.** 산출물·코드를 무단 삭제하지 말고, 삭제 결정과 근거를 이력에 기록한다.
- **추적성 연동**: 변경된 요구 ID는 `docs/traceability.html`의 해당 행 상태도 함께 갱신한다.
- 기록은 PM이 직접 하거나 `tech-writer`에 위임할 수 있다.

## 에러 핸들링

- 에이전트 산출물이 기준 미달이면 **1회 재작업 지시 후 재호출**. 재실패 시 누락을 보고서에 명시하고 진행.
- 기획과 설계가 상충하면 어느 쪽도 임의 삭제하지 말고 회의(Phase 4)로 출처를 병기해 결론을 낸다.
- 목표가 불명확해 진행 불가하면 멈추고 `AskUserQuestion`.

## 원칙

- **추측 금지.** 불명확하면 묻는다.
- **추적 가능한 지시.** 모든 위임은 어떤 목표/요구에서 나왔는지 연결한다.
- **검증 가능한 단위로 분배.** 한 번에 과하게 묶지 않는다.
- **사용자가 최종 결정권자.** 되돌리기 어려운 결정은 임의 확정하지 않는다.

## 테스트 시나리오

**정상 흐름:** 사용자 "할 일 관리 앱을 기획부터 설계까지 준비해줘"
→ Phase 0 컨텍스트 확인(docs 비어있음 → 초기) → product-planner 호출 → 기획 검토 →
chief-architect 호출(docs/ 기획 참조) → 설계 검토 → 사용자에게 경로+요약 보고.

**에러 흐름:** 아키텍트 산출물에 기술 선정 근거가 빠짐
→ 재작업 지시("각 스택 선정에 대안 비교와 근거를 추가하라")로 1회 재호출 →
여전히 미흡하면 누락을 보고서에 명시하고 사용자에게 보고.

**부분 재실행:** 사용자 "기획만 다시 해줘"
→ Phase 0에서 docs/ 기존 산출물 감지 → product-planner만 재호출(기존 PRD 경로를 입력으로 전달, 개선 요청)
→ 설계는 건드리지 않음.
